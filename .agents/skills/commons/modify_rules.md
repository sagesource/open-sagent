## 重要说明

**⚠️ 警告：在修改项目的任何代码和方案之前，必须先阅读并遵循本文档的约束。**

**任何涉及方案和代码的变更，只有经过用户（项目 Owner）评审通过后才可实施。**

**严禁未经评审直接修改任何工程代码文件。**

---

## 方案评审流程

### 1. 生成方案文档

- 在 .agents/plan 目录下创建方案 Markdown 文件
- 文件名格式：`[序号]-[简要描述].md`，如：`001-添加提现审核功能.md`
- 序号从 001 开始递增

### 2. 方案内容要求

- 修改背景和目的
- 详细修改方案（含文件变更列表）
- 影响范围分析
- 测试计划

### 3. 等待评审

- **只有经过评审并确认后的方案，才允许合并到工程中**
- 在方案文件中添加评审记录章节
- **评审结论说明**：
    - ✅ **通过**：用户明确表示方案通过，可实施代码变更
    - ❌ **需修改/未通过**：用户提出修改意见或改进建议，表示方案未通过

### 4. 评审通过后的处理

- 方案评审通过后，**必须在方案文件的「评审记录」表格中更新评审结论为「通过」**，并填写评审人和时间
- 完成评审记录更新后，方可实施代码变更

### 5. 评审未通过的处理

- 如果评审人提出修改意见（表示未通过）：
    1. **在原方案文件上修改**：根据评审意见更新方案内容，不创建新方案文件
    2. **保留历史记录**：保留原有方案内容，通过修订记录体现变更
    3. **更新评审记录**：在方案文件的「评审记录」表格中追加新的评审记录
    4. **重新提交评审**：修改完成后，再次提交评审
- ⚠️ **禁止行为**：评审未通过时创建新的方案文件替代原有方案

### 6. 禁止行为

- ❌ 未经评审直接修改工程代码
- ❌ 评审未通过时创建新的方案文件替代原有方案
- ❌ 在方案文件中使用覆盖整个文件的 `WriteFile` 操作
- ❌ 删除或修改已有的方案文件

### 7. 已评审通过方案的后续变更

- 已评审通过的方案如需修改（如代码实现调整、bug 修复、细节优化等），**必须在对应方案文件中记录变更历史**
- 在方案文档中添加「方案变更记录」章节，记录以下内容：
    - 变更时间
    - 变更原因
    - 变更的文件列表
    - 关键代码变更对比（修改前 vs 修改后）
- 示例：
    ```markdown
        ## 方案变更记录
        ### 变更 1（2026-03-31）：修复 refId 赋值和 JSON 序列化问题

        **变更原因：**
        1. refId 应使用 requestId 而非 refAccountId
        2. DTO 流式访问器导致 Fastjson 无法正确序列化
     
        **文件变更：**
  
        | 文件路径 | 变更类型 | 说明 |
        |----------|----------|------|
        | `XxxDTO.java` | 修改 | 移除 `@Accessors(fluent = true)`，新增 `toJsonObject()` 方法 |
     
        **关键代码变更：**
        ```java
            // 修改前
            xxx.refId(detail.getId());
            // 修改后
            xxx.refId(dto.getRequestId());
        ```
    ```
  
## 方案文档段落结构约束

为维护方案文档的可读性和一致性，**必须严格保持以下段落结构**，各章节职责单一，禁止互相穿插：

1. **修改方案章节 只展示最终状态**
    - `文件变更列表` 和 `详细变更内容` 必须反映**当前已评审通过后的最终代码状态**
    - 禁止在修改方案章节中夹杂历史变更过程（如"修改前 vs 修改后"的对比）

2. **所有历史变更必须且只能集中在「方案变更记录」章节**
    - 已评审通过的方案如需后续调整，必须在统一的「方案变更记录」章节中按时间顺序记录
    - 每个变更条目应包含：变更时间、变更原因、文件变更列表、关键代码变更对比
    - **禁止**在文档其他位置（如评审记录章节、测试计划章节、修改方案章节）插入变更记录

3. **所有评审结论必须且只能集中在「评审记录」章节**
    - 评审记录以表格形式呈现，包含评审人、时间、结论、备注四列
    - **禁止**在评审记录表格中或表格前后插入任何变更内容、代码片段或方案说明
    - **禁止**将评审结论分散写入各个变更条目之后

4. **章节顺序固定**
   ```
   1. 背景与目的
   2. 修改方案
   3. 影响范围分析
   4. 测试计划
   5. 方案变更记录（如存在后续变更）
   6. 评审记录
   ```
- 不得颠倒章节顺序，不得在固定章节之间插入新的同级章节
- 示例
```markdown
# 方案：[序号]-[简要描述]

## 1. 背景与目的

## 2. 修改方案

| 文件路径 | 变更类型 | 说明 |
|----------|----------|------|
| `xxx.java` | 新增/修改/删除 | 说明 |

### 2.2 详细变更内容

#### 文件 1: xxx.java

    ```java
    // 新增/修改的代码片段
    ```

## 3. 影响范围分析

## 4. 测试计划

## 5. 评审记录

| 评审人 | 时间 | 结论 | 备注 |
|--------|------|------|------|
| [待填写] | [待填写] | [通过/需修改] | |
```
